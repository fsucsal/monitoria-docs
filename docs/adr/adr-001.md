# ADR-001 — Estilo arquitetural: monólito modular com OOP bem definido

**Status:** Aprovado

## Contexto

O sistema será desenvolvido por uma equipe acadêmica, majoritariamente iniciante, com um escopo bem definido e sem necessidade imediata de alta escalabilidade distribuída.

Existe a necessidade de:

- Manter **boa separação de responsabilidades**
- Garantir **baixo acoplamento**
- Evitar **complexidade arquitetural desnecessária**
- Favorecer a **manutenibilidade, testabilidade e aprendizado**
- Permitir **evolução futura**, caso o projeto cresça

Além disso, a equipe utilizará os recursos nativos do framework para:

- Validações
- Injeção de dependência
- Gerenciamento transacional

## Decisão

Adotar uma **Arquitetura de Monólito Modular com Orientação a Objetos bem definida**, com os seguintes princípios:

- ✅ O sistema será um **monólito único**
- ✅ A organização será feita por **módulos funcionais**
- ✅ Cada módulo terá:
  - Controller
  - Service
  - Repository
  - DTOs próprios
- ✅ As **regras de negócio residem exclusivamente nos Services**
- ✅ As **entidades serão anêmicas**, com uso de:
  - Validações via anotações do framework
  - Sem lógica de negócio nas entidades no início
- ✅ Não será aplicado **DDD completo**
- ✅ Poderá ser aplicado um **DDD Light**, apenas no nível de:
  - Linguagem ubíqua
  - Separação conceitual de módulos
- ✅ A comunicação entre módulos ocorrerá via:
  - Chamadas de Service
  - Contratos explícitos

O objetivo principal é manter a arquitetura:

> **Simples o suficiente para não travar o time, mas estruturada o bastante para não virar um monolito caótico.**

## Alternativas Consideradas

- Microsserviços desde o início
- DDD completo com entidades ricas
- Monólito tradicional sem modularização
- Arquitetura hexagonal completa desde a base

Essas alternativas foram descartadas por adicionarem **complexidade excessiva para o momento do projeto e para o perfil do time**.

## Consequências

### Positivas

- ✅ Arquitetura simples de entender
- ✅ Facilita onboarding de desenvolvedores iniciantes
- ✅ Boa organização por responsabilidade
- ✅ Baixo custo operacional
- ✅ Facilidade para testes unitários e de integração
- ✅ Facilita futura migração para microsserviços, se necessário
- ✅ Evita overengineering no início do projeto

### Negativas

- ❌ Menor flexibilidade para escalabilidade independente no curto prazo
- ❌ Entidades não concentram comportamento no modelo atual
- ❌ Parte da expressividade de domínio é reduzida sem DDD completo
- ❌ Evoluções arquiteturais mais profundas exigirão refatorações futuras

## Diretrizes Arquiteturais

- Controllers:

  - Apenas orquestram requisição e resposta
  - Não contêm regras de negócio

- Services:

  - Centralizam toda a lógica de negócio
  - Devem ser pequenos, coesos e testáveis

- Repositories:

  - Apenas acesso a dados
  - Nenhuma regra de negócio

- DTOs:

  - Usados para entrada e saída de dados
  - Validados via anotações

- Entidades:
  - Representam apenas o estado
  - Sem regras de negócio no início
