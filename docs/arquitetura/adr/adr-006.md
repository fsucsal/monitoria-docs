# ADR-006 — Containerização: Docker como padrão de execução

**Status:** Aprovado

## Contexto

O sistema será desenvolvido por uma equipe acadêmica, composta por desenvolvedores com diferentes níveis de experiência e utilizando diferentes sistemas operacionais e ambientes locais.

Além disso:

- Não há um profissional dedicado exclusivamente a DevOps
- O projeto contará com **DevOps em formação**
- A stack tecnológica definida inclui:

  - Backend em Spring
  - Frontend em React (ou Angular)
  - Banco de dados PostgreSQL

- É necessário garantir:

  - Padronização total do ambiente
  - Facilidade para subir o projeto localmente
  - Eliminação de problemas de “funciona na minha máquina”
  - Onboarding rápido e previsível de novos membros

Dado esse contexto, o ambiente de execução precisa ser simples, reproduzível e confiável.

## Decisão

Padronizar o uso do **Docker** como tecnologia oficial de **containerização e execução do sistema**, adotando os seguintes princípios:

- Toda a aplicação deverá rodar em containers
- O banco de dados deverá rodar obrigatoriamente em container
- O frontend poderá rodar:

  - Preferencialmente em container
  - Ou localmente em modo de desenvolvimento, quando necessário

- Será utilizado **docker-compose** para orquestrar:

  - Backend
  - Frontend
  - Banco de dados

- O ambiente Docker será tratado como:

  - Padrão oficial de desenvolvimento
  - Base para homologação
  - Base para produção

O Docker será considerado a **fonte oficial da verdade do ambiente de execução da aplicação**.

## Alternativas Consideradas

- Execução direta no host (sem containerização)
- Uso de máquinas virtuais tradicionais
- Ambientes configurados manualmente, sem padronização

Essas alternativas foram descartadas por:

- Gerarem inconsistências entre ambientes
- Aumentarem o esforço de configuração manual
- Dificultarem o onboarding de novos membros
- Elevaram o risco de erros difíceis de reproduzir
- Não prepararem o projeto para automação futura

## Consequências

### Consequências Positivas

- Ambiente totalmente reproduzível
- Redução significativa de problemas de configuração
- Facilidade para automação de build e deploy no futuro
- Padronização entre todos os desenvolvedores
- Base sólida para adoção futura de CI/CD
- Forte valor educacional para o time

### Consequências Negativas

- Curva de aprendizado inicial para membros iniciantes
- Dependência de máquinas capazes de executar containers
- Possível aumento no consumo de recursos locais

## Diretrizes de Uso

- O repositório do projeto deverá conter, no mínimo:

  - `Dockerfile` para o backend
  - `Dockerfile` para o frontend (quando aplicável)
  - `docker-compose.yml` para execução completa do sistema

- Nenhum desenvolvedor deverá:

  - Subir ambientes de homologação ou produção fora do Docker
  - Definir dependências apenas no ambiente local, fora dos containers

- As configurações de ambiente devem:

  - Ser declaradas via variáveis de ambiente
  - Estar documentadas no repositório

- A equipe de **DevOps em formação** será responsável por:

  - Manter os arquivos Docker atualizados
  - Garantir que o ambiente descrito no `docker-compose` reflita o ambiente real de execução
  - Apoiar o time na resolução de problemas relacionados ao ambiente
