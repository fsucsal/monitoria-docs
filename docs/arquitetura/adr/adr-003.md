# ADR-003 — Banco de Dados Principal: PostgreSQL

**Status:** Aprovado

## Contexto

O sistema será desenvolvido em um contexto acadêmico, utilizando:

- Backend em Spring
- Frontend em React
- Arquitetura de Monólito Modular

O domínio da aplicação exige forte **integridade relacional** entre os dados, envolvendo entidades como:

- Usuários e perfis de acesso
- Cursos, turmas e disciplinas
- Inscrições, vínculos e processos seletivos
- Registros operacionais e históricos acadêmicos

Além disso, o projeto precisa atender aos seguintes requisitos não funcionais:

- Simplicidade de manutenção
- Baixo custo operacional
- Facilidade de ensino para desenvolvedores iniciantes
- Consistência e confiabilidade dos dados
- Possibilidade de evolução futura da arquitetura e do modelo de dados

Dado o escopo atual do projeto, não há necessidade de soluções distribuídas ou bancos especializados desde o início.

## Decisão

Adotar o **PostgreSQL** como **banco de dados relacional principal** da aplicação.

A utilização do PostgreSQL seguirá os seguintes princípios:

- Modelo de dados relacional e normalizado
- Uso explícito de chaves primárias e estrangeiras
- Garantia de integridade referencial no banco
- Uso de transações ACID para operações críticas
- Controle de versão do esquema de dados por meio de migrations
- Utilização de um único banco de dados principal na fase inicial do projeto

O PostgreSQL será tratado como a **fonte oficial da verdade dos dados** do sistema.

## Alternativas Consideradas

- MySQL
- MongoDB
- SQL Server
- Uso combinado de múltiplos bancos desde o início

Essas alternativas foram descartadas pelos seguintes motivos:

- Aumento desnecessário da complexidade operacional
- Dificuldade maior no ensino de modelagem e integridade relacional
- Ausência de benefícios claros frente ao escopo atual do projeto
- Maior custo cognitivo para equipes iniciantes

## Consequências

### Consequências Positivas

- Forte suporte a relacionamentos complexos
- Garantia de integridade e consistência dos dados
- Ampla maturidade e estabilidade da tecnologia
- Excelente integração com o ecossistema Spring
- Facilidade de ensino de conceitos fundamentais de bancos relacionais
- Suporte a evoluções futuras do modelo de dados sem ruptura arquitetural

### Consequências Negativas

- Menor flexibilidade para armazenar dados altamente não estruturados
- Possível necessidade de otimizações futuras em cenários de alto volume
- Dependência de migrations bem controladas para evitar inconsistências

## Diretrizes de Uso

- Todo acesso a dados deve ocorrer via Repositories
- Nenhuma regra de negócio deve residir no banco
- Constraints de integridade devem ser utilizadas sempre que possível
- Alterações no esquema devem ser feitas exclusivamente via migrations versionadas
- Consultas complexas devem ser justificadas e documentadas
