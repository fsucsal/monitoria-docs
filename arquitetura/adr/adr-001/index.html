    ADR-001 — Estilo Arquitetural: Monólito Modular

    Status: Aprovado

    Contexto

    O Sistema de Seleção de Monitoria — UCSal será desenvolvido em ambiente acadêmico, por um time majoritariamente iniciante, porém com a exigência de qualidade próxima a produção.

    As principais necessidades identificadas foram:

    • Separação clara de responsabilidades
    • Baixo acoplamento entre partes do sistema
    • Facilidade de aprendizado e onboarding
    • Alta legibilidade e previsibilidade do código
    • Suporte a testes unitários como base
    • Evolução futura sem bloqueios arquiteturais

    O projeto não possui, neste momento:

    • Requisitos de escalabilidade distribuída
    • Necessidade de deploys independentes
    • Complexidade de domínio que justifique DDD completo

    Além disso, a stack escolhida (Java + Spring Boot) oferece recursos maduros para:

    • Injeção de dependência
    • Gerenciamento transacional
    • Validações declarativas
    • Tratamento centralizado de exceções

    Decisão

    Foi decidido adotar uma Arquitetura de Monólito Modular, organizada por módulos funcionais, com forte uso de Orientação a Objetos tradicional e padrões consagrados do ecossistema Spring.

    As diretrizes da decisão são:

    • O sistema será um único artefato implantável (monólito)
    • A organização do código será feita por módulos, e não por camadas globais
    • Cada módulo é responsável por um contexto funcional claro

    Cada módulo deverá conter internamente:

    • Controllers (exposição REST)
    • Services (regras de negócio)
    • Repositories (acesso a dados)
    • DTOs (entrada e saída da API)
    • Entidades JPA

    Regras estruturais importantes

    • As regras de negócio residem exclusivamente nos Services
    • Controllers não contêm lógica de negócio
    • Repositories não contêm lógica de negócio
    • Entidades JPA representam estado e persistência
    • Validações de negócio ocorrem nos Services
    • Validações de estrutura de dados ocorrem nos DTOs (Bean Validation)

    Sobre DDD

    • DDD completo não será adotado
    • Não serão utilizadas entidades ricas, aggregates ou domain services

    • Será adotado apenas um DDD leve, limitado a:

    • Linguagem ubíqua

    • Separação conceitual de módulos

    Essa escolha está alinhada com os documentos de:

    • Organização dos Módulos
    • Guia de Padrões de Desenvolvimento Backend
    • Estrutura do Projeto

    Alternativas Consideradas

    As seguintes alternativas foram avaliadas e descartadas:

    Microsserviços desde o início

    • Complexidade operacional elevada
    • Necessidade de infraestrutura inexistente no contexto atual
    • Overhead cognitivo alto para o time

    DDD completo com entidades ricas

    • Curva de aprendizado elevada
    • Alto risco de uso incorreto dos conceitos
    • Maior custo de manutenção inicial

    Monólito tradicional sem modularização

    • Alto risco de acoplamento
    • Dificuldade de evolução
    • Baixa previsibilidade estrutural

    Arquitetura Hexagonal completa

    • Excesso de abstrações para o escopo atual
    • Ganho técnico baixo frente à complexidade adicionada

    Consequências

    Consequências Positivas

    • Arquitetura simples e previsível
    • Facilita onboarding de desenvolvedores iniciantes
    • Código mais fácil de testar e manter
    • Redução de overengineering
    • Boa base para evolução futura
    • Compatível com boas práticas de mercado

    Consequências Negativas

    • Menor expressividade de domínio sem DDD completo
    • Entidades com baixo comportamento
    • Possível necessidade de refatoração caso o sistema evolua significativamente

    Essas consequências são consideradas aceitáveis frente ao contexto do projeto.

    Diretrizes Arquiteturais Derivadas

    Este ADR estabelece as seguintes diretrizes obrigatórias:

    • Controllers

    • Apenas recebem e respondem requisições

    • Não contêm regra de negócio

    • Services

    • Centralizam regras de negócio

    • Devem ser coesos, testáveis e explícitos

    • Repositories

    • Responsáveis apenas por persistência

    • DTOs

    • Utilizados como contrato da API

    • Validados via Bean Validation

    • Entidades JPA

    • Representam o modelo persistente

    • Não concentram regras de negócio

    Este ADR serve como base para todas as demais decisões arquiteturais do projeto. Alterações devem ser discutidas e registradas em novos ADRs.
